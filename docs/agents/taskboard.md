# Taskboard

## Current Objective

Add per-user idol record persistence to the idols page, using the existing WeChat-login user session as the user identity and Cloudflare Worker D1/R2 storage for structured records and avatars.

Scope constraints:

- Use WeChat login as the formal user identity mechanism. Frontend calls `wx.login()` and sends the returned code to Backend; Backend exchanges it with WeChat `code2Session`, maps `openid` to an internal `user_id`, and issues a Chekinana user session token.
- WeChat login must be completely independent from the existing backend token authentication. WeChat login/session endpoints must not require `X-Cheki-Token` or any scanner access token.
- The existing backend token authentication is only for entering and using the scanner/extraction workflow. Other mini-program functions must not require that token.
- Scanner/extraction APIs must keep their existing token protection unless explicitly changed later. Non-scanner functions should not become scanner-token-gated as part of WeChat login.
- WeChat login success must be clearly visible in the Settings page, including enough non-sensitive state for the user to know they are logged in.
- Backend must keep WeChat `appid`, `appsecret`, any session signing secret, and WeChat session data server-side only. Do not put these values in mini-program code, docs, handoffs, logs, or git.
- Backend should treat `openid` as the stable per-mini-program identity. `unionid` may be stored when returned, but implementation must not require it because it may be absent.
- Backend must not send `session_key` to Frontend. It is server-side material only.
- Backend should expose a clear login/session contract and a way for protected user-data APIs to derive `user_id` from the user session token.
- Frontend should provide the WeChat-login entry in Settings only. The existing `auth` page is not an app-wide authentication page; it is the legacy scanner-token entry used only before entering the scanner/extraction workflow.
- Do not add or keep a WeChat-login button, WeChat-login status, or WeChat-login side effect on the existing `auth` page. The `auth` and `index` names are historical from when the mini program only had the scanner feature.
- The `auth` page must not prefill `chekinana-preview`, must not auto-authenticate, and must not automatically enter `index`. Users enter the scanner only after manually typing a real scanner token or `chekinana-preview` and tapping the authentication button.
- The `api.chekinana.top/*` Cloudflare Worker must host scanner-independent services directly, or through non-RunPod Cloudflare-managed bindings, so they continue working when no RunPod pod is running. WeChat login/session verification/contact must not depend on scanner token, pod id, or any RunPod Flask backend.
- Contact email must retain the existing Cloudflare Worker Email Sending path and add a Gmail API delivery path. Cloudflare Email Sending may be used when Workers Paid/Email Sending is enabled; Gmail API should provide a non-RunPod HTTPS delivery path that can run without upgrading to Workers Paid.
- Contact-author UI relocation to Settings and Add Idol dialog UI were approved in reviewer commit `4c8a105`; this round builds on that UI rather than reopening it.
- Saving idol records requires WeChat login. Missing/invalid user session should prompt the user to log in and must not fall back to scanner token authentication.
- Store idol records per user with `idol_name`, `group_name`, avatar, selected colors, `weibo_uid`, `weibo_username`, and `weibo_verification`.
- Search result cards always display the newly searched Weibo avatar, username, and verification. Stored same-uid records with changed username or verification must render as separate stored-record cards, not overwrite the search result.
- Use D1 for structured idol records and R2 for stored idol avatars. The mini program must not hold Cloudflare write credentials or write R2 directly.
- Scanner/extraction APIs still require the existing scanner token and RunPod pod id routing.
- Existing scanner, extraction, result download, contact email, lianliankan gameplay, settings hidden routes, Weibo OpenAPI lookup, RunPod startup, pod ID scanner-token flow, and Cloudflare asset routes must not regress.

Contact email contract:

- `POST /api/contact` keeps the existing route, auth, rate limit, and response shape.
- Optional `contact` must be included in the email body when provided and omitted when empty.
- Backend must log successful contact email sends as well as failures.
